关于git的学习记录——基础篇 oio

学习网站：Learn Git Branching

Git 希望提交记录尽可能地轻量，因此在你每次进行提交时，它并不会盲目地复制整个目录。条件允许的情况下，它会将当前版本与仓库中的上一个版本进行对比，并把所有的差异打包到一起作为一个提交记录。

Git 主要命令

Git Commit：提交新记录，添加新节点

Git Branch <name>：基于这个提交及他所有的parent新建分支仅新建分支

main*代表你当前在main分支下

git checkout <name>：切换分支

创建一个新的分支同时切换到新创建的分支的话，可以通过 git checkout -b <your-branch-name>

新版本使用git switch [] [--no-guess]

git merge <your-branch-name> 合并分支

git rebase <your-branch-name> [your-rebase-name]线性合并分支
Rebase 实际上就是取出一系列的提交记录，“复制”它们，然后在另外一个地方逐个的放下去。
Rebase 的优势就是可以创造更线性的提交历史。

Git 特性

关于HEAD变量:

通过git chekout <name>命令可以切换HEAD的指向

可以HEAD→main →C4（main→C4）
也可以HEAD→C4

相对引用——^,~

git log 查看提交记录的哈希值

其哈希值可以只输入前几个字母进行匹配
但是这样太麻烦了

^: 使用 ^ 向上移动 1 个提交记录 (父节点，且可叠用。HEAD^,main^^)

当^后面加数字时，表示的是移动到他的第几个parent（当我们使用git merge命令时可能会有一个以上的parent）

~: 使用 ~<num> 向上移动多个提交记录，如 ~3,HEAD~3

两个符号可以链式操作，例如git checkout HEAD~^2~2
结合之前的命令实用：
git branch -f main HEAD~3：将 main 分支强制指向 HEAD 的第 3 级 parent 提交。

撤销变更 —— git reset和git revert

git reset <参数>
在reset后， 撤销部分所做的变更还在，但是处于未加入暂存区状态。本地常用
git revert <参数>
把撤销之前的部分作为新分支提交远程常用

整理提交记录

复制git cherry-pick

git cherry-pick <提交号>...
将一些提交复制到当前所在的位置（HEAD）下面

交互式的 rebase

git rebase -i HEAD~4
会有一个交互式的页面供你进行排序、删除、合并
在你完成后，会将处理好的分支放在当前分支下

只取一个提交记录（实战结合）

通过以上两个命令把修改bug中最后解决bug的提交记录复制或移动到main分支下

提交的技巧

当你希望对之前的提交记录进行小小的修改，这里有两个方法。

git commit --amend追加提交
将不会重复提交而是修改之前的提交于同一位置

可以先移动顺序，将要修改的放前面来，修改后再排序放回去。最后合并main分支。

使用git cherry-pick一步步复制修改

Git tag —— 标签/锚点

git tag <标签名> <指向位置>
可以通过git check移动到标签

git describe 可以 描述离你最近的锚点（也就是标签)

语法：git describe <ref>^1
返回值:<tag>-<numCommits>-g<hash>^2

  1. <ref> 可以是任何能被 Git 识别成提交记录的引用，如果你没有指定的话，Git 会使用你目前所在的位置（HEAD）。

  2. tag 表示的是离 ref 最近的标签， 
    numCommits 是表示这个 ref 与 tag 相差有多少个提交记录，
    hash 表示的是你所给定的 ref 所表示的提交记录哈希值的前几位。

多分支 rebase

git rebase <1> <2>1→2
2分支中所有的提交记录都将合并到1分支之下
